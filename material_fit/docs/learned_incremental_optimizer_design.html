Final design proposal

跨引擎材质对齐：学习型增量优化器最终设计

本文给出一个理论上完整的最终版方案：用真实 Laya 渲染作为黑盒 oracle， 用多视角渲染误差作为反馈，训练一个带语义先验、历史记忆和候选选择能力的 learned incremental optimizer，而不是训练一个一次性参数翻译器。

项目：material_fit 日期：2026-06-15 核心路线：POMDP + RL/L2O + Surrogate + True Render Validation

1. 总体结论

最终系统应该被定义为 Render-Feedback Conditioned Learned Optimizer for Cross-Engine Material Alignment。 它不是把 Unity 材质参数直接翻译成 Laya 参数的模型，也不是一个纯 CMA-ES 或纯启发式优化器。 它是一个在真实渲染反馈下运行的学习型增量优化系统。

一句话定义： 给定 Unity 参考渲染、多视角相机、Laya shader / .lmat / 贴图和初始 Laya 渲染， 系统在每一步观察当前视觉残差和历史动作反馈，选择下一批最值得真实渲染的参数候选， 最终输出历史最佳 .lmat，并给出可解释的失败或收敛诊断。

这个设计接受一个基本事实：跨引擎材质对齐没有可靠的全局闭式解。 Unity 的视觉结果可能不在 Laya shader 的表达空间内；同一外观可能对应多组 Laya 参数； 贴图通道、灯光、色彩空间、后处理和 shader keyword 分支都会改变最优解。因此最终方案不追求 Unity 参数 -> Laya 参数 的静态映射，而追求 视觉误差 + 历史反馈 + shader 语义 -> 下一步最有效的参数更新。

最终形态应具备的能力

真实渲染闭环

所有最终判断以 Laya 真实渲染为准。模型和 surrogate 只能加速候选生成，不能替代最终验证。

学习型增量更新

策略输出下一步动作：调哪个语义组、哪些参数、方向、步长和候选批，而不是一次吐最终参数。

可解释诊断

输出不只是 best .lmat，还要说明每轮为什么动这些参数，以及失败是优化不足还是 shader 表达力不足。

2. 问题定义

当前任务可以形式化为一个昂贵黑盒优化问题。我们有一个目标渲染集合 Y = {I_unity(view_i)}，以及一个参数化的 Laya 渲染器 R_Laya(theta, view_i)。目标是找到一组 Laya 材质参数 theta*，使多视角 Laya 渲染在视觉上接近 Unity 参考。 

theta* = argmax_theta Score(
  { I_unity(view_i) },
  { R_Laya(theta, view_i) }
)

subject to:
  theta belongs to valid .lmat parameter space
  shader gates / defines / bounds are respected
  texture bindings and render states are not corrupted
  final candidate must be verified by true Laya render

为什么不是普通监督学习

如果我们有大量成对数据 (Unity material, ideal Laya .lmat)， 可以训练一个参数回归模型。但现实里“ideal Laya 参数”并不唯一，而且通常没有标注。 更严重的是，即使标注存在，它也只对特定 shader、特定资产、特定灯光配置成立。 因此直接监督学习的泛化性很弱。

为什么不是纯 RL 环境

这个任务可以写成 RL，但它不是天然的游戏式 MDP。当前图像残差并不能完整描述 loss landscape。 哪些参数有效、哪些参数互相耦合、哪个方向过去试过没用，这些信息都需要从历史动作和反馈中估计。 所以更准确的建模是 POMDP + belief-state optimizer。

关键判断： 最优设计不是让 RL 直接控制 50 维连续参数，而是让它学习一个分层优化策略： 先决定“在哪里搜”，再决定“怎么搜”，最后用真实 Laya 渲染验证候选。

3. 参考工作启发

下面这些工作共同支持本设计的核心选择：使用渲染空间 loss、采用迭代增量更新、 在黑盒不可微环境中用 RL 或 learned optimizer 提高样本效率，并保留真实渲染验证。

方向 代表工作 对本项目的启发
RL for procedural material Procedural Material Generation with Reinforcement Learning 不可微程序材质也可以用图像相似度做反馈；RL 不需要端到端可微渲染器。 这直接对应 Laya 黑盒渲染场景。
Learned gradient descent Single-Image SVBRDF Estimation with Learned Gradient Descent 材质反问题适合“迭代预测更新量”。这支持我们做 learned delta / learned optimizer， 而不是一次性输出最终 .lmat。
Learning to optimize Learning to Optimize, MetaBO, B2Opt 优化器本身可以被学习，尤其是在一类相关黑盒任务上。我们的“同类 shader / 同类资产” 正是可迁移结构的来源。
Differentiable material graph DiffMat, MATch 材质优化的正确目标应在渲染外观空间，而不是参数空间。即使我们没有可微 Laya， 也应该沿用 render-space loss。
Rendering-aware inverse rendering Single-Image SVBRDF Capture with a Rendering-Aware Deep Network loss 应该跨视角/光照评估，因为材质参数本身不唯一；渲染后的视觉一致性才是最终目标。

4. 设计原则

  1. 真实渲染是最高裁判。 Surrogate、神经策略、CMA-ES 和启发式都只能提出候选。候选是否有效必须由真实 Laya PNG 与 Unity PNG 的多视角对比决定。
  2. 模型学习“怎么优化”，不是学习“唯一答案”。 训练目标是减少达到同等视觉质量所需的真实渲染次数，而不是证明存在唯一参数映射。
  3. 分层动作空间优先于全维连续控制。 高层选择语义组和策略模式，低层选择参数、方向、步长和候选分布。这样更稳定、更可解释。
  4. 多视角最差项必须进入目标。 平均分提升但某个角度崩掉，在生产上不可接受。reward 和模型 value 都必须看 worst view。
  5. 安全层不可被模型绕过。 模型不能直接写 .lmat。所有输出必须经过 bounds、shape、gate、reserved-key 和 no-corruption 检查。
  6. 失败诊断是产品能力的一部分。 如果 shader 表达空间不包含目标效果，系统应该尽早停止并报告缺失能力，而不是无限调参。

5. 最终系统架构

最终架构分为四层：任务规范化层、渲染与评分层、学习型优化层、训练与数据闭环层。 当前仓库已经具备其中一部分：.lmat 安全写入、多视角 Laya capture、图像评分、CMA-ES 和语义优化器。 最终版是在此基础上加入 trajectory dataset、surrogate、learned policy 和离线/在线训练。

Task Canonicalizer 统一 Unity/Laya 输入、相机、灯光、贴图、shader metadata。
Semantic Effect Graph 参数分组、gate、bounds、role、transform、搜索优先级。
Observation Builder 融合图像残差、通道指标、历史动作、best state。
Learned Optimizer 输出语义组、参数 mask、delta 分布、候选评分、停止诊断。
True Laya Oracle 真实应用 .lmat 并同视角渲染 PNG，作为最终反馈。
输入规范化 语义约束 构造 belief 产生候选 真实验证
Trajectory Store 保存每一步 params、图像、分数、动作、reward 和诊断。
Surrogate Model 快速预测候选 score/residual/uncertainty，减少真实渲染。
Candidate Generator 混合 policy samples、CMA-ES、BO、semantic probes。
Safety Layer .lmat shape guard、bounds、gate、reserved key、回滚。
Training Pipeline 行为克隆、离线 RL、model-based RL、在线持续改进。

闭环关系

真实 Laya 渲染负责产生高质量但昂贵的反馈；surrogate 负责在真实渲染前预筛大量候选； learned optimizer 负责根据历史和语义决定下一步搜索方向；trajectory store 则把每次生产运行转化为训练数据。 这使系统具有自我改进能力：跑得越多，对同类 shader 和资产的搜索越快。

6. POMDP 建模

标准 MDP 假设当前 state 足以预测下一步转移和奖励。但在本问题中，当前图像和当前参数并不能完全揭示 shader 的响应曲面。例如某个参数当前无效，可能是因为 gate 没打开；某组参数单独无效但组合有效； 某个方向过去已经证明会破坏最差视角。这些信息来自历史。因此最终建模应是 POMDP： policy 维护一个由历史轨迹推断出的 belief state。

Observation

o_t = {
  current_params,
  initial_params,
  best_params,
  semantic_effect_graph,
  per_view_reference_images,
  per_view_candidate_images,
  residual_maps,
  material_channel_metrics,
  research_metrics,
  adjustment_hints,
  last_k_actions,
  last_k_rewards,
  response_map,
  renderer_metadata
}

Hidden State

隐状态是不可直接观测的 loss landscape：每个参数的局部方向、参数耦合、gate 关系、shader 表达能力上限、 渲染噪声和 metric 漏洞。模型通过 history encoder、response map 和 surrogate uncertainty 估计这个隐状态。

Action

a_t = {
  high_level:
    strategy_mode: semantic_probe | cma_subspace | bo_batch | exploit_best | stop
    semantic_group: base_color | shadow | specular | fresnel | emission | reflection | color_grade
    budget_allocation: n_candidates, n_true_renders

  low_level:
    param_mask,
    direction_vector,
    step_size,
    covariance_or_temperature,
    candidate_batch
}

Transition

Transition 不是由模型完全掌控，而是由真实 Laya 渲染器和评分系统定义： 写入候选参数，渲染多视角 PNG，计算多视角分数和通道误差，再更新历史。

Reward

Reward 应该同时鼓励整体提升、最差视角提升、关键 bottleneck 缓解和低成本搜索，同时惩罚无效渲染、 大幅破坏、越界、无意义参数抖动和指标漏洞。

7. 核心模块

Task Canonicalizer
把 Unity 参考、Laya 工程、相机、灯光、模型、贴图、材质路径和 shader metadata 规整为一个可复现实验任务。 它的输出必须足够完整，使任务可以在任意机器上重放。
Semantic Graph
记录参数所属语义组、数值范围、是否可搜索、gate 条件、依赖关系和风险等级。 它可以由规则、LLM 语义预分析和人工校正共同产生，但最终必须是结构化、可校验的。
Renderer Oracle
接收参数候选，返回同视角 Laya PNG、mask、alpha、渲染日志和失败状态。 最终版应该是常驻服务，而不是每轮启动编辑器或依赖桌面截图。
Observation Builder
把当前图像残差、多视角指标、通道 bias、语义图、历史动作和 best state 编码成 policy 输入。 它是把渲染问题转成学习问题的关键边界。
Learned Optimizer
分层 policy。高层选择语义组和搜索策略，低层输出参数 mask、方向、步长和候选分布。 它还输出 expected improvement、uncertainty 和 stop/failure 判断。
Candidate Generator
把 learned policy 输出转成实际候选。候选来源包括 policy samples、CMA-ES 分布、Bayesian acquisition、 semantic probes、archive restart 和局部 best refinement。
Safety Layer
任何候选写入前必须经过安全层：参数是否存在、shape 是否一致、是否越界、是否触碰 reserved key、 gate 是否合理、是否需要回滚。模型不能绕过它。
Surrogate Model
学习 (asset context, params, view) 到 score/residual/uncertainty 的近似。 它用于候选预筛和 model-based RL，但最终不能替代真实渲染验收。
Trajectory Store
保存所有任务、候选、渲染、指标、动作和奖励。它既是 UI forensic 数据，也是后续训练数据。

8. 模型结构

最终模型应是多输入、多头输出的分层策略网络。它同时看图像、参数、语义图和历史， 输出高层搜索策略、低层参数更新和候选价值估计。

Residual Encoder 编码多视角参考图、候选图、差分图、mask、最差视角。
Param Encoder 编码初始、当前、best 参数以及归一化后的 delta。
Graph Encoder 编码 shader semantic graph、gate、依赖和参数组。
History Encoder 编码最近动作、reward、response map 和失败样本。
Fusion Transformer 形成 belief state，供多个决策头共享。
图像残差 参数状态 shader 语义 历史反馈 融合表征
Group Head 选择 base_color/specular/fresnel 等语义组。
Param Mask Head 选择具体参数或参数组合。
Delta Head 输出方向、步长、连续 delta 分布。
Acquisition Head 预测候选 expected improvement 和不确定性。
Stop/Failure Head 判断继续、收敛、配置错误或 shader 表达能力不足。

为什么不用单个回归头

单个回归头直接输出完整参数向量，会把反问题的非唯一性强行压成单答案； 同时也很难解释为什么某个参数被改。分层策略更符合生产需求：它能告诉我们当前主要问题是什么， 为什么选择某个语义组，以及这次候选预期改善哪个通道。

输出形式

policy_output = {
  group_probs,
  strategy_mode_probs,
  param_mask_logits,
  delta_mean,
  delta_scale_or_covariance,
  candidate_expected_improvement,
  candidate_uncertainty,
  stop_probability,
  failure_reason_probs
}

9. 奖励与指标

奖励函数不能只看 fit_score 的绝对值。绝对分数会受 case 难度影响，且容易让策略偏向简单样本。 更好的训练信号是增量改善、最差视角改善和 bottleneck channel 改善。 

reward_t =
  1.00 * delta_aggregate_score
+ 0.60 * delta_worst_view_score
+ 0.40 * delta_bottleneck_channel_score
+ 0.20 * accepted_visual_margin
- 0.10 * normalized_param_change
- 0.05 * render_cost
- invalid_penalty
- regression_penalty

Reward 组成解释

项 目的 原因
delta_aggregate_score 提升整体视觉接近度 主优化目标，来自多视角聚合分数。
delta_worst_view_score 保护最差角度 避免平均分提升但某个视角严重退化。
delta_bottleneck_channel_score 缓解当前最大视觉问题 鼓励策略针对颜色、高光、阴影、Fresnel 等具体问题做有效动作。
normalized_param_change 抑制无意义大幅跳动 同等视觉改善下，参数改动越小越稳定、越可解释。
render_cost 提高样本效率 真实 Laya 渲染昂贵，策略必须学会少渲染。
invalid_penalty 避免非法候选 越界、shape 错误、gate 错误、渲染失败都应被强惩罚。

最终验收指标

  • Best final score：固定真实渲染预算下的历史最佳多视角分数。
  • Time-to-quality：达到某个视觉阈值所需真实渲染次数。
  • Worst-view recovery：最差视角是否同步改善。
  • Human accept rate：人工或验收规则是否接受最终效果。
  • Failure classification accuracy：系统是否正确判断 shader 不可表达、配置错误或继续优化仍有空间。

10. 训练体系

最终训练不是单阶段完成，而是从可控合成任务、历史优化轨迹、surrogate、离线 RL 到在线自改进逐步建立。 即使不考虑成本，分阶段训练仍然是理论上更稳的设计，因为它能分离“会不会看懂残差”和“会不会做长期决策”。

1. Synthetic Recovery 随机采样 Laya 参数生成目标，再从扰动初值恢复。
2. Behavior Cloning 模仿 heuristic、CMA-ES、人工成功轨迹。
3. Surrogate Training 学习参数到 score/residual/uncertainty 的近似。
4. Offline RL 从已收集轨迹学习长期收益策略。
5. Online Validation 真实 Laya 小批量验证，持续写回轨迹库。

Stage 1：合成自监督恢复

在 Laya 内部随机采样合法参数 theta_target，渲染得到目标图； 再从扰动后的 theta_init 开始，训练策略恢复 theta_target 的外观。 这不需要 Unity 标注，可以大规模生成。注意训练目标仍然是 render-space similarity， 而不是参数 MSE，因为参数非唯一。

Stage 2：行为克隆

用当前启发式、CMA-ES、semantic_group、人类调参记录生成 expert-like trajectories。 行为克隆让 policy 先学会基本常识：颜色偏亮该往哪调，高光过强该试哪些参数， gate 没开时不要浪费预算。

Stage 3：Surrogate 训练

Surrogate 输入资产上下文、参数、视角和当前 residual，输出候选分数、残差预测和不确定性。 训练时必须用真实渲染轨迹监督。最终 surrogate 不直接决定结果，只用于候选预筛和 model-based RL。

Stage 4：离线 RL

从历史轨迹训练 policy，使其最大化长期 reward。适合使用 conservative/offline RL 思路， 避免策略在数据分布外输出危险候选。离线 RL 的目标是超过 imitation：学会在不同 case 下分配预算， 学会何时探索组合参数，何时回到 best branch。

Stage 5：Model-based RL + 真渲染校正

在 surrogate 中进行大量虚拟 rollout，然后用真实 Laya oracle 抽样验证。 真实验证结果再反哺 surrogate 和 policy。该机制让系统随着生产运行持续改进。

11. 推理闭环

最终推理不是单次前向，而是一个带候选批、surrogate 预筛和真实渲染验证的循环。 

initialize task
render initial Laya views
score initial state
archive initial state as best

while budget remains:
  obs = build_observation(current, best, history, residuals, semantic_graph)
  policy_output = learned_optimizer(obs)
  candidate_pool = generate_candidates(policy_output, cma_es, bo, semantic_probes)
  valid_candidates = safety_filter(candidate_pool)
  surrogate_ranked = surrogate_prescreen(valid_candidates)
  selected = choose_top_k_with_uncertainty(surrogate_ranked)
  true_results = laya_oracle.render(selected)
  metrics = score_multiview(true_results)
  update_history(metrics, actions, rewards)
  update_best_if_improved(metrics)
  if stop_head says converged or impossible:
    break

restore best .lmat
write report and failure diagnosis

候选选择策略

每轮不要只渲染一个候选。最终版应生成几十到几百个候选，用 surrogate 预筛后真实渲染 top-k。 top-k 选择不应只看预测均值，还要看不确定性：高不确定但潜在收益高的候选可以作为探索项。

停止条件

  • 达到目标分数或人工验收阈值。
  • 最近若干真实渲染没有显著提升，且 surrogate 也预测继续收益低。
  • 最差视角不再改善，且候选主要在参数空间内震荡。
  • failure head 判断 shader 表达能力不足，例如缺失 rim/fresnel、matcap、emission 或关键 mask 通道。
  • 渲染环境异常，例如 Laya 未刷新、相机不一致、背景/mask 不可靠。

12. 数据与日志

Trajectory store 是最终系统的资产。没有高质量轨迹，就没有泛化的 learned optimizer。 每个真实渲染候选都要被结构化保存，以便复盘、训练和 benchmark。

为什么必须保存失败样本

失败样本告诉模型什么不要做。对于黑盒优化，负样本和无效动作同样重要： 它们能帮助 policy 学会 gate、参数冷却、no-effect 参数识别、以及何时不要继续浪费真实渲染预算。

13. 评价体系

最终系统必须在多个泛化层级上评估，而不是只看单个资产跑得好不好。

评估层级 问题 指标
Seen asset / seen shader 同资产重复优化是否更快？ 渲染次数、最终分数、收敛稳定性。
Unseen asset / seen shader 同 shader 家族是否能迁移？ time-to-quality、best score、最差视角改善。
Seen family / modified shader 新增参数或 gate 后是否仍可工作？ 安全失败率、参数选择准确性、fallback 质量。
Unseen shader family 模型能否退化成合理 baseline？ 不应显著差于 CMA-ES/semantic_group；失败诊断应可靠。
Human acceptance 视觉上是否可交付？ 人工 A/B、盲测、验收通过率。

必须保留的 baseline

  • 当前 heuristic。
  • CMA-ES cold。
  • CMA-ES warm。
  • CMA-ES + hint bias。
  • semantic_group / adaptive_response_search。
  • 纯 surrogate 选择但无 learned policy。
  • learned policy 无 surrogate。
  • learned policy + surrogate + true-render validation 完整系统。

15. 项目路线

下面不是成本压缩版，而是理论上完整系统应走的路线。每一步都服务于最终闭环。

Phase A：任务和轨迹标准化

  • 定义 canonical task package：Unity reference、Laya project、shader metadata、camera/light protocol。
  • 定义 trajectory schema：episode、step、candidate、render、metric、reward、diagnostic。
  • 把当前 decision.json / iteration_series.json 升级成训练可用数据格式。
  • 所有历史 run 迁移或转换为统一 schema。

Phase B：真实渲染服务化

  • 实现常驻 Laya render worker，支持参数 patch、批量候选、多视角 RenderTexture 输出。
  • 去掉对桌面截图的主路径依赖。
  • 加入 render determinism check：同参数多次渲染的噪声范围必须可量化。

Phase C：大规模数据生成

  • 合成 recovery tasks：随机合法参数作为 target，从扰动初值恢复。
  • 真实 cross-engine tasks：Unity reference 到 Laya initial 的真实对齐。
  • 采集 baseline 轨迹：heuristic、CMA-ES、semantic_group、human edits。
  • 系统性保存失败样本和 no-effect actions。

Phase D：Surrogate 和不确定性模型

  • 训练 score surrogate：预测 aggregate/worst/channel 分数。
  • 训练 residual surrogate：预测差分图或低维 residual embedding。
  • 输出 uncertainty，用于候选选择和 active learning。
  • 建立 surrogate calibration benchmark，防止 surrogate 过度自信。

Phase E：学习型增量优化器

  • 先用 behavior cloning 学会 baseline 行为。
  • 再用 offline RL 学长期 reward。
  • 最后用 model-based RL 在 surrogate 上做虚拟 rollout，并定期真渲染校正。
  • 训练 stop/failure head，输出不可表达或配置错误诊断。

Phase F：完整系统集成

  • Policy + candidate generator + surrogate + true render oracle 联合推理。
  • UI 显示每轮策略意图、候选来源、预期收益、真实收益和失败诊断。
  • 生产运行自动追加到 trajectory store，形成持续学习闭环。

Phase G：跨资产/跨 shader 基准

  • 建立 seen/unseen asset、seen/unseen shader、不同灯光/背景的 benchmark。
  • 固定真实渲染预算，比较所有 baseline。
  • 输出论文/报告级结果：sample efficiency、final quality、failure diagnosis、ablation。

16. 风险边界

1. 不存在全局闭解

最终系统不能承诺任意 Unity shader 都能转换成任意 Laya shader。它能做的是在给定 Laya 表达空间内更快找到好近似， 并在不可表达时给出诊断。

2. Reward hacking

如果评分函数对背景、mask 或抗锯齿有漏洞，policy 会利用这些漏洞。因此 reward 必须混合多视角、最差视角、 通道指标、结构指标和人工验收信号，并持续做 metric validation。

3. Surrogate 过度自信

Surrogate 如果在分布外预测错误，会把搜索带偏。因此候选选择必须保留不确定性探索， 并要求最终候选由真实 Laya 渲染验证。

4. 泛化边界

泛化主要存在于同类 shader、同类资产和同类渲染协议中。跨 shader 家族时， 系统应该能安全退化到 CMA-ES / semantic_group，而不是强行相信 learned policy。

5. 数据偏差

如果训练数据只来自少数资产或少数材质类型，policy 会学习到偏置。最终 benchmark 必须显式覆盖不同材质、 不同贴图分布、不同光照和不同 shader 分支。
